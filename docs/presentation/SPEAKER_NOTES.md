
#### [2:35~2:50] Content Analyzer 6단계
```
"Content Analyzer는 6단계를 거칩니다.

1단계: 사용자의 원문에서 감정을 분류합니다.
2단계: 내러티브 구조를 파악합니다. '시작-갈등-현재' 같은 형태로요.

4단계: 핵심 인사이트를 추출합니다. 사용자가 정말 말하고 싶은 게 뭔가?
5단계: 대상 감정 상태를 식별합니다. 어떤 심리 상태여야 도움이 될까?
6단계: 이 모든 정보를 다음 에이전트에게 넘깁니다.

이렇게 하면 단순한 요약이 아니라,
구조화된 심리 분석이 됩니다."
```

**시각화 팁**:

- 번호를 가리키며 설명

#### [2:50~3:05] Podcast Reasoning의 추론 경로
```
"그 다음 Podcast Reasoning이 들어옵니다.

여기서는 Tree of Thought 패턴을 사용합니다.
하나의 응답만 생성하지 않고,
최대 3개의 다른 응답 경로를 동시에 생각해봅니다.

예를 들어:
경로 A: 사용자의 감정에 공감하고 현재를 인정해주기
경로 B: 그 감정의 원인을 함께 탐색하기
경로 C: 작은 변화부터 시작할 수 있도록 희망 제시하기

그 다음 이 경로들이 Neo4j에 그래프로 저장됩니다.
나중에 '이 사용자는 어떤 방식의 조언을 선호하나?'
를 분석할 수 있게요.

또한 조건에 따라 Episode Memory와 Knowledge Agent를
호출할지 판단합니다.

'이 사용자와 비슷한 상황의 과거 대화가 있나?'
'이게 전문 지식이 필요한 영역인가?'
하는 식으로요."
```

#### [3:05~3:20] Script Generator
```
"마지막으로 Script Generator가
이 모든 정보를 바탕으로 팟캐스트 스크립트를 생성합니다.

단순한 텍스트 생성이 아닙니다.
'공감-전환-희망' 세 부분의 구조로 고정되어 있습니다.

공감: '당신의 감정은 타당하다'
전환: '하지만 여기서 바꿀 수 있는 부분이 있다'
희망: '이렇게 하면 좀 나아질 수 있다'

이 구조가 심리학적으로 효과가 입증되었기 때문입니다.

생성된 스크립트는 일관되고 설득력 있게 들립니다."
```

### Task 2: 감정 시각화 (3:20~3:44) — 개발자2 담당

#### [3:20~3:28] 감정 → 이미지 변환 로직
```
"두 번째 기술은 감정의 시각화입니다.

Emotion Agent가 추출한 벡터를 봅시다:
- primary_emotion (감정명)
- intensity (강도, 0~1)
- valence (긍정/부정도, -1~1)
- arousal (활성화 정도, 0~1)

이 4개의 수치를 가지고,
자연어 프롬프트로 변환합니다.

예: 'sadness, intensity 0.8, valence -0.6, arousal 0.3'
→ 'soft blue tones, rainfall, solitary figure looking out at horizon, hope breaking through clouds'"
```

**시각화**:
- Slide 7: 왼쪽에는 감정 벡터, 오른쪽에는 생성된 이미지
- 화살표로 변환 과정 표시

#### [3:28~3:36] Stable Diffusion + LoRA
```
"그 다음 Stable Diffusion에 통과됩니다.
Stable Diffusion은 고품질 이미지 생성 모델입니다.

더 정교한 결과를 위해 LoRA(Low-Rank Adaptation)라는
파인튜닝 기법을 사용했습니다.

일반적인 Stable Diffusion이 아니라,
'감정을 시각화하는 특화된' 모델로 훈련되었다는 뜻입니다.

추가로 사용자의 나이, 선호 스타일을 반영합니다.
20대 사용자는 밝고 현대적인 톤,
50대 사용자는 부드럽고 서정적인 톤으로요."
```

**강조**:
- "특화된" 이라는 단어 강조
- LoRA의 개념을 최대한 단순하게 설명

#### [3:36~3:44] 저장과 활용
```
"생성된 이미지는 S3에 저장되고,
CDN을 통해 프론트엔드에 전달됩니다.

사용자는 팟캐스트 에피소드를 볼 때,
감정에 맞는 커버 이미지도 함께 봅니다.

텍스트와 이미지가 함께 작용하면서,
'따뜻한 감성'이라는 우리의 핵심 가치가
전달됩니다."
```

**마무리**:

### Task 3: 할루시네이션 방지 (3:44~4:10) — 개발자3 담당

#### [3:44~3:52] Batch Validator의 역할
```
"세 번째는 AI가 거짓말(할루시네이션)을 하지 못하도록
막는 검증 시스템입니다.

LLM은 가끔 그럴듯해 보이지만 사실이 아닌 것을
생성합니다. 우리는 이를 방지하기 위해
Batch Validator라는 에이전트를 만들었습니다.

이것은 '자체 반사(Self-Reflection)' 패턴입니다.
LLM이 자신이 생성한 것을 다시 검토하는 것이죠."
```

#### [3:52~4:02] 5가지 검증 기준
```
"Batch Validator는 생성된 스크립트를 
5가지 기준으로 재평가합니다.

첫째, 감정 상태 일관성. 사용자가 불안해하는데
스크립트가 너무 느긋한 톤이면 안 됩니다.

둘째, 구조 완전성. '공감-전환-희망' 구조가 맞는가?

셋째, 안전성 준수. 의료 조언은 없는가? 위험한 내용은?

넷째, 개인화 품질. 이 사용자의 특성이 반영됐는가?

다섯째, 톤/스타일 매칭. 사용자의 성격과 맞는가?

각 기준별로 0~1 점수를 받고, 평균을 냅니다."
```

**시각화**:
- Slide 8: 5개 기준을 시각적으로 표시
- 점수 계산 과정을 보여줌 (예: 0.92, 0.85, 1.0... → 평균 0.88)

#### [4:02~4:10] 재시도 로직
```
"만약 점수가 0.75 이상이면 통과입니다.

하지만 0.5~0.75 사이면?
Script Generator에게 다시 만들도록 요청합니다.
```

### Q4: 데이터 개인정보는 어떻게 보호하나?
```
A (개발자2):
"3가지 층위에서 보호합니다.

1층: 입력 정제 (InputSanitizer)
   프롬프트 인젝션 공격을 방어합니다.
   예: '; DROP TABLE users; --' 같은 악의적 입력을 탐지

2층: 처리 중 암호화
   Backend API 통신은 HTTPS (TLS 1.3)
   민감한 설정은 AWS KMS로 관리

3층: 출력 정제 (OutputSanitizer)
   생성된 응답에서 PII (전화번호, 주민번호, 주소)를
   자동으로 마스킹합니다.

또한 사용자 데이터는 사용자의 동의 하에만
Episode Memory에 저장됩니다."
```

## 발표 팁 & 팀 협력

### 발표 자세
- 청중과 눈 맞춤: 매 10~15초마다 전체 청중을 훑기
- 중립적 표정 유지: 과하지 않은 미소
- 손동작: 자연스럽고, 설명할 때만 사용
- 이동: 한 자리에 서기 (떠돌아다니지 않기)

### 발표 속도
- 평소보다 15~20% 느리게
- 각 문장 끝에 0.5초 멈춤
- 중요한 수치는 2번 반복

### 팀 협력
- **핸드오프**: "그 다음 개발자2가 설명하겠습니다"
- **에러 발생 시**: 다른 팀원이 백업 설명 준비
- **시간 초과**: 개발자1이 감지하고 신호 (손가락으로 "1분 남음")
- **질문 중간에**: 일단 발표자가 시작하되, 어려우면 다른 개발자에게 손으로 요청

### 리허설 일정
- 최종 리허설: 발표 하루 전 (전체 5분 통과)
- 개별 준비: 각자 담당 부분 3회 이상 연습
- 데모 테스트: 발표 직전 3회 실행 (응답 시간 측정)
